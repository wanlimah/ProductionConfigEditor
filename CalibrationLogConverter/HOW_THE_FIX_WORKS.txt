════════════════════════════════════════════════════════════════════════════════
WHY DID UseHeaderRow=true LIMIT ROWS?
════════════════════════════════════════════════════════════════════════════════

This appears to be a limitation or bug in the ExcelDataReader library when
processing certain Excel file formats. The exact cause is unclear, but the
behavior is consistent:

- UseHeaderRow=true → Approximately 183 rows read (varies by file)
- UseHeaderRow=false → ALL rows read

The solution: Don't rely on the library to handle headers. Do it manually.

WHAT IF I STILL GET 183 RECORDS?
════════════════════════════════════════════════════════════════════════════════

Possible reasons:

1. Running old executable
   Solution: Make sure you're running the NEW executable at:
   CalibrationLogConverter\bin\Release\net8.0-windows\CalibrationLogConverter.exe

2. Source file actually has only 183 rows
   Solution: Open your Excel file and check the actual row count

3. Many rows are being skipped (empty or invalid)
   Solution: Check debug output to see skip reasons

4. Multiple files, only one being processed
   Solution: Check that all files are in Raw_Data folder
